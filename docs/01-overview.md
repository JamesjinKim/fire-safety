## 1. 문제 정의

영세한 한국 소방 점검업체는 다음 일을 매번 손으로 한다.

- 건물마다 건축물대장(건축년도·용도·면적·층수) → **그 시점 적용 소방법령** → 적용 소방시설 리스트 산출.
- **별지 제9호서식(자체점검 결과보고서)** + **별지 제4호서식(점검표)** 양식 작성·인쇄·종이 제출.
- 사무실에서 가정한 시설 리스트와 **현장 실제가 다르면** 즉시 수정·근거 기록.

기존 통합 ERP SaaS(라이팅넷 SGuard+S-CES, 세정아이앤씨)는 **무겁고 비싸 영세 점검업체의 회피 1순위**.
시장 점유율이 낮은 진짜 이유는 솔루션 깊이가 아니라 **무게**와 **종이행정 잔존**. 빈 자리가 명확하다.

## 2. 해결 방식

**Light SaaS — "9대 의무 다 안 합니다. 자체점검 일지·보고서만은 한국에서 가장 정확하고 가장 가볍게."**

- Next.js 웹앱 단일 인터페이스. 모든 계산·매칭은 웹앱 내부 함수가 결정론적으로 수행.
- v1 핵심 흐름 (단계 A · 사무실 관리자):
  1. 건축물대장 입력 → 법제처 OpenAPI로 그 시점 법령 조회 → 적용 시설·점검표 산출
  2. **작업지시 문서 동결** (immutable hash, 변조 감지)
  3. 별지 9호/4호 HTML 렌더 → A4 1:1 인쇄 → 종이 제출
  4. 청구는 세금계산서 PDF로 떨어뜨림 (v1.5에서 ASP 자동 발행)
- v2 핵심 흐름 (단계 B · 현장 점검자, 다중 협업): 작업지시 기준 점검 + **Discrepancy 변경 로그**(append-only).
- v1 PoC 산출물은 단계 A E2E + 단계 B를 위한 데이터모델 자리만 확보.

**핵심 원칙: 계산·판정은 코드가, LLM 호출은 0.** (환각 차단)

자세한 시장 진단·차별 포지셔닝·파트너 운영 원칙은 메모리 `market-strategy` 참조.

## 3. 의사결정 이력 (현 결정에 영향 주는 것만)

### 3.1 (2026-05-21) RAG 폐기
초기에 "로컬 LLM + 법령 벡터검색(RAG)"이 검토됐으나 **폐기**. 본 도메인 본질은 자유 질의응답이 아니라 정밀 룰 계산. RAG는 숫자 기준 환각 위험.

### 3.2 (2026-05-21) Greenfield (fork 안 함)
참고했던 건설안전 OSS(agent-safety-oss)를 **fork하지 않고 새로 만든다.** 소방 등급 체계가 건설보다 정밀하여 전용 룰 엔진을 깨끗하게 짜는 게 환각 차단에 유리. 설계 패턴(SSoT 마스터 JSON·문서 노드 스키마)은 차용. 코드 결합도 실측 등 상세는 archive 참조.

### 3.3 (2026-05-23 → 05-24) MCP의 위상 재정의 → v2 보류
- 초기 의도: "Claude Desktop이 호출하는 MCP 도구 + 현장 음성 웹앱"
- 페르소나가 **LLM 호출자 → 사무실 웹폼 사용자**로 이동하면서 v1 흐름에 LLM 호스트가 끼어드는 지점이 사실상 없음.
- 결정: **v1은 웹앱 일원화**. MCP 후보 도메인 4함수(`lookupLawByYear` 외)는 `apps/web/lib/domain/`에 순수 함수로 구현. v2에서 "Claude Desktop에서 같은 도구 호출" 실수요가 검증되면 얇은 MCP 래퍼로 부활.
- 보류 자료: [`_archive/2026-05-21/`](_archive/2026-05-21/).

### 3.4 (2026-05-23) Feature 범위 피벗
정적 데이터 SSoT 단독 feature(`fire-duty-master`)에서 **상위 워크플로 feature(`fire-inspection-system`)**로 재편성. `fire-duty-master.json`은 하위 데이터셋으로 격하.

### 3.5 (2026-05-23) 두 단계 워크플로 분리
사용자 통찰로 도메인 모델 자체를 두 단계로 분리:
- **단계 A** 산출: `Project` → `WorkOrder` (immutable, hash)
- **단계 B** 결과: `Inspection` → `Discrepancy` (append-only)
- 법제처 API 응답속도 우려가 이 분리로 해소 (단계 A만 동기 호출, 단계 B는 네트워크 의존 0).

### 3.6 (2026-05-24) 시장 진단·전략 박제
경쟁사 분석(라이팅넷 SGuard+S-CES, 세정) 후 **"Light SaaS · 좁고 깊게 → SaaS 확장"** 포지셔닝 확정. 20년 베테랑 공동개발 파트너 확보. 9대 의무 통합 ERP는 영구 배제(점검업체 회피 1순위). 상세는 `market-strategy` 메모리.
