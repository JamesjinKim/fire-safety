PRD v0.1 · 운영 아키텍처 반영

건축물대장에서 법정 서식까지 이어지는 소방안전점검 Light SaaS

건물주 위탁 요청을 시작점으로, 건축물대장 OCR 검수, 법정 요건 기반 설비 추천, 설비별 현장 입력, 별지 제4호·제9호 출력, 비용 PDF 생성을 하나의 프로젝트 흐름으로 묶는다.

제품의 첫인상

관리자가 건축물대장을 업로드하면 OCR 초안이 생성되고, 확정된 건물 정보로 점검 설비 섹션이 추천된다. 현장은 추천·동결된 설비 섹션을 기준으로 결과만 입력한다.

1 프로젝트 단위로 고객, 건물, 작업지시, 점검, 출력, 비용을 묶음
2 관리자와 현장 사용자가 모두 로그인하는 v1 구조
4/9 별지 제4호 점검표와 제9호 결과보고서를 핵심 산출물로 고정
A4 법정 양식 HTML을 PDF와 인쇄물로 저장하는 출력 중심 설계

확정된 제품 방향

이번 PRD의 기준점이다. 자동화는 입력 부담을 줄이되, 법정 판단에는 관리자 확정값만 사용한다.

01

OCR은 초안

건축물대장 PDF/이미지를 읽지만, 관리자가 필수 필드를 확정하기 전에는 설비 추천을 막는다.

02

설비 자동 추천

건축년도, 용도, 면적, 층수 등 확정값을 기준으로 법정 요건에 맞는 설비 섹션을 추천한다.

03

관리자 동결

추천 결과는 관리자가 포함, 제외, 수정 후 작업지시로 동결한다. 현장 차이는 변경 로그로 남긴다.

04

현장은 설비별

점검 입력 단위는 별지 제4호 구조에 맞춰 소화기구, 옥내소화전, 유도등 같은 설비 섹션으로 나눈다.

05

런타임 분리

고객 관리자가 자기 API 키로 법령 스냅샷을 만들고, 설비 추천은 우리 룰 엔진이 수행한다.

엔드투엔드 흐름

고객사 위탁 요청부터 비용 PDF까지 이어지는 v1 업무 흐름. 회색 지점은 시스템 자동화, 노란 지점은 관리자 검수 게이트다.

STEP 1

고객사·건물 등록

건물주 또는 관리주체, 점검 대상 건물을 프로젝트에 연결한다.

STEP 2

건축물대장 업로드

PDF 또는 이미지를 업로드하고 OCR 초안을 생성한다.

STEP 3

필수 필드 확정

원본과 추출값을 비교해 사용승인일, 용도, 면적, 층수 등을 확정한다.

STEP 4

설비 섹션 추천

법정 요건 DB와 룰 엔진으로 적용 설비를 추천한다.

STEP 5

작업지시 동결

관리자가 추천 결과를 검수하고 현장용 설비 섹션을 확정한다.

STEP 6

현장 입력·출력

설비별 점검 결과를 입력하고 별지 제4호·제9호와 비용 PDF를 생성한다.

차단 게이트

OCR 필수 필드가 미확정이면 설비 추천 버튼은 비활성화된다.

검수 게이트

시스템 추천 설비는 바로 작업지시가 되지 않고 관리자 검토를 거친다.

동결 게이트

작업지시는 해시와 함께 저장되며, 현장 변경은 별도 로그로 추적한다.

웹서버와 법령 API 운영 구조

Vercel 런타임에는 법제처 API 키를 두지 않는다. 고객 관리자 세션에서 필요한 법령 스냅샷만 생성하고, 추천 판단은 우리 룰 엔진이 맡는다.

사용자

관리자·현장 사용자

브라우저에서 프로젝트 생성, OCR 검수, 설비 추천 검토, 현장 입력, 출력 확인을 수행한다.

Web Runtime

Vercel 우선 배포

Next.js App Router 앱을 icn1 Seoul에 두고, 불가 시 hnd1 Tokyo 또는 sin1 Singapore를 사용한다.

Data

Supabase Seoul

PostgreSQL, Auth, Storage를 ap-northeast-2에 둔다. 첨부와 생성 PDF는 Vercel 파일시스템에 저장하지 않는다.

BYOK Collection

고객 키 기반 수집

고객 관리자가 입력한 법제처 API 키로 필요한 시행일 스냅샷을 생성한다. 키는 저장하지 않는다.

Official API

국가법령정보 API

eflaw와 eflawjosub로 시행일 기준 본문·조문을 가져와 검증 스냅샷으로 승격한다.

사용자 요청 경로

브라우저 → Vercel → Supabase snapshot → 룰 엔진. 서버의 법제처 API 키는 0개다.

법령 수집 경로

고객 관리자 세션 → 법제처 API → snapshot JSON → Supabase 저장.

환경변수 경계

Vercel은 LAW_DATA_MODE=runtime만 사용하고 법제처 API 키는 등록하지 않는다.

실패 정책

고객이 만드는 것은 법령 스냅샷이다. 설비 리스트는 우리 룰 엔진이 생성한다.

OCR 검수 화면의 역할

OCR은 편의 기능이지만, 제품 신뢰는 검수 UX에서 결정된다. 관리자는 원본과 추출값을 나란히 보며 확정한다.

왼쪽: 원본 문서

건축물대장 PDF 또는 이미지 원본을 표시한다. 페이지 이동, 확대, 축소, 필드 위치 확인이 필요하다.

  • PDF/이미지 원본 확인
  • 추출 필드의 근거 위치 확인
  • 저신뢰 필드 빠른 검토

오른쪽: 확정 입력 폼

추출값은 초안으로 들어오며, 관리자가 확인해야 확정값으로 바뀐다. 설비 추천은 확정값만 사용한다.

  • 추출됨, 수정됨, 누락, 저신뢰, 확정 상태 표시
  • 필수 필드 미확정 목록 표시
  • 확정 완료 후 설비 추천 실행

사용자와 권한

v1은 관리자와 현장 사용자를 모두 로그인 대상으로 둔다. 팀장/점검자 세분화는 나중에 확장한다.

관리자

프로젝트 책임자

고객사, 건물, OCR 검수, 설비 추천 검토, 작업지시 발행, 최종 출력, 비용 PDF 생성을 담당한다.

현장 사용자

설비별 입력자

배정된 작업지시를 확인하고 설비별 점검 결과, 불량 내용, 메모, 사진, 음성 파일을 입력한다.

고객사

데이터 대상

v1에서는 포털 사용자가 아니라 점검 의뢰자, 건물주, 공급받는자 정보로 관리한다.

핵심 객체 지도

객체는 업무 흐름을 그대로 반영한다. 나중에 협업, AI 정리, 정산 시스템으로 확장할 수 있도록 경계를 분리한다.

v1 범위와 후속 범위

제품 매력을 위해 OCR과 현장 입력은 v1에 포함한다. AI 정리, 전자발행, 협업은 다음 단계로 분리한다.

v1에 포함
  • 관리자와 현장 사용자 로그인
  • 건축물대장 PDF/이미지 업로드
  • OCR 초안 생성 및 관리자 확정
  • 법정 요건 기반 설비 섹션 자동 추천
  • 관리자 검수 후 작업지시 동결
  • 설비별 현장 점검 결과 입력
  • 텍스트, 사진, 음성, 파일 기록 저장
  • 별지 제4호·제9호 HTML/PDF 출력
  • 관리자 입력 기반 세금계산서 형태 PDF
v1 이후로 분리
  • 음성 인식과 AI 문장 정리
  • Google Drive 자동 저장
  • 실제 국세청 전자세금계산서 발행
  • 정산·회계 시스템
  • 고객사 포털
  • 다중 점검자 실시간 협업
  • 소방청 제출 시스템 연동
  • 9대 의무 통합 ERP 기능

법정 서식 연결 방식

현장에서 한 번 입력한 설비별 결과가 별지 제4호 점검표와 별지 제9호 불량 세부사항으로 동시에 연결된다.

별지 제4호

설비별 점검표가 원천 데이터다. 점검번호 체계는 원문과 같은 형태로 보존한다.

  • 예: 1-A-007
  • 결과: 적합, 부적합, 해당없음, 미점검
  • 사진과 메모는 점검항목 또는 설비 섹션에 연결

별지 제9호

건물 정보와 점검 결과 요약, 불량 세부사항을 프로젝트 데이터에서 구성한다.

  • 건축물 정보는 관리자 확정값 사용
  • 설비 현황은 작업지시와 현장 결과에서 구성
  • 부적합 항목은 불량 세부사항으로 자동 집계
