## 1. 배경 / 문제

기존 `fire-duty-master` feature는 **법정문서 마스터 JSON(SSoT)** 작성이 목표였다. 그러나
2026-05-23 도착한 현장 요구사항은 다음과 같은 **end-to-end 워크플로**였다.

1. 건축물대장(건축년도·용도·면적) → 적용 소방시설 리스트와 점검 범위 자동 산출
2. 건축년도에 해당하는 **그 시점의 소방법령**을 매칭 (1980~현재 개정 시계열)
3. 매칭 결과로 **별지 제9호서식**(자체점검 결과보고서) + **별지 제4호서식**(점검표) 작성
4. A4(210×297) 인쇄 1:1 동일 사이즈로 HTML 양식 출력
5. 여러 점검자가 현장에서 각자 부분 작성 → **하나의 문서로 통합**

→ 마스터 JSON만으로는 가치 0. 상위 워크플로 feature로 재정의한다.
→ `fire-duty-master`는 본 feature가 참조하는 **하위 데이터셋**으로 격하한다.

## 2. 목표 (이 feature의 범위)

Next.js 웹앱 단일 인터페이스의 소방안전점검 워크플로 PoC를 구축한다.
**PoC 1단계는 단일 사용자·단일 건축물 기준 E2E 양식 완성까지**를 목표로 한다 (사용자 결정).

> 2026-05-24 갱신: 초기 계획의 MCP 서버는 v2 보류. 도메인 로직은 `apps/web/lib/domain/`의
> 순수 함수로 구현하여 v2 MCP 부활 시 얇은 래퍼로 노출 가능하게 설계. 사유: [../../01-overview.md §3.3](../../01-overview.md).

### 2.1 핵심 사용자 흐름 (PoC v1)

```
[입력] 건축물대장 항목 (건축년도, 용도, 연면적, 층수, 다중이용업소 여부 등)
   ↓
[엔진] 건축년도 → 그 시점 소방시설법/화재예방법 매칭
   ↓
[산출] 해당 건축물에 적용되는 소방시설 리스트 + 점검대상 (별지 4호 32종 중 해당분)
   ↓
[양식] 별지 제9호서식 (8쪽) + 별지 제4호서식 (해당 점검표만) HTML 생성
   ↓
[출력] 화면 표시 + A4 인쇄 1:1 PDF
```

### 2.2 PoC 1단계 산출물

- `apps/web/`: Next.js 웹앱 (단일 건축물 입력 폼 → 양식 미리보기 → 인쇄)
- `apps/web/lib/domain/`: 4함수 (`lookupLawByYear`, `deriveInspectionScope`, `composeWorkOrder`, `validateInspection`) — v2 MCP 래핑 대비 순수 함수 형태
- `data/fire-duty-master.json`: 기존 산출물, 본 feature가 참조
- `data/inspection-checklist.json`: 별지 제4호 32종 점검항목 마스터 (PoC는 우선 5~7종)
- `data/law-snapshots/`: 법제처 API 응답 캐시 (빌드타임 스냅샷 폴백)
- 양식 HTML 컴포넌트: `별지9호.tsx`, `별지4호.tsx` (A4 210×297mm 정확 출력)
- **세금계산서 PDF 출력** (v1.5에서 ASP 자동 발행 전환): 점검 완료 후 청구 PDF 떨어뜨림. 사용자가 외부 시스템에서 직접 발행.
- **셀프 온보딩 흐름**: 가입·구독 결제·첫 점검 일지 생성까지 SI 개입 0으로 가능 (KPI: 가입 3분, 첫 보고서 30분)

### 2.3 범위 밖 (단계별 분리)

**v1.5 후보**
- **세금계산서 ASP 자동 발행** (세로이디·더존·가비아 등 API 연동, 공동인증서·국세청 e-세로)
- 청구·정산 흐름 (이행계획서 자동 생성)

**v2 후보**
- **다중 점검자 실시간 협업/문서 통합** — 아키텍처는 v1에서 미리 설계, 구현은 v2 (sub-feature: `inspection-collab`)
- **연도별 법령 시계열 풀 코퍼스 (1980~)** — v1은 최근 5년 + 주요 개정시점(2003/2012/2024) 우선
- 별표 제10호 이행계획서, 점검인력 배치확인서
- NFPC/NFTC 기술기준 본문 번들 (nfsc.go.kr 수집)
- 보고서 제출(소방본부장 송신)·전자정부 연동
- 음성 입력·현장 모드 (단계 B 부활 후보)
- **MCP 래퍼** (`apps/mcp/`) — `apps/web/lib/domain/` 4함수를 Claude Desktop에 노출

**v3 후보**
- 인력 관리·일정 관리

**명시적 영구 배제 (당분간)**
- 전자결재 다단계 (라이팅넷·세정 따라가지 않음 — 영세 점검업체 회피 1순위)
- 회계 분개·재무제표
- 인사·근태·급여
- 재고 관리
- 9대 의무 통합 ERP화 (B 전략 — 시장 실패 사례)

### 2.4 차별 포지셔닝 (한 줄 정의)

> "9대 의무 다 안 합니다. **자체점검 일지·보고서**만은 한국에서 가장 정확하고 가장 가볍게 합니다.
> 가입은 3분, 구독은 셀프, 점검 한 건은 30분."

라이팅넷·세정의 "통합 ERP SaaS"와 의도적으로 **다른 시장**을 본다. 상세는 [[market-strategy]] 메모리 참조.

### 2.5 공동개발 파트너 (자산)

20년 베테랑 점검업체 1곳과 PoC 공동개발. 주 1회 정기 리뷰로 단계별 검증.
v1 완성 직후 **2번째 파트너**(다른 규모·지역) 일찍 확보 — 한 회사 입맛 함정 회피.
파트너 의견은 절대치가 아니라 **검증 데이터의 하나**로 다룬다 (베테랑 함정 회피).

## 3. 기술 결정 (사용자 확정)

| 항목 | 결정 | 이유 / 보완 |
|---|---|---|
| Feature 구조 | 상위 `fire-inspection-system` 신규, `fire-duty-master`는 하위 데이터셋 | 워크플로 단위가 사용자 가치의 단위 |

| 실시간 협업 (v2) | bkend.ai BaaS + 섹션(설비) 단위 락킹 | CRDT보다 구현 난이도 낮음, Dynamic 레벨에 적합 |

| 프로젝트 레벨 | **Dynamic** | bkend BaaS 도입 + 협업 v2 예정으로 Starter 불가 |

## 4. 핵심 아키텍처 개략 (Design에서 확정)

```
┌─────────────────────────────────────────────────┐
│  Next.js Web App (apps/web)                     │
│  ─ 건축물대장 입력 폼 (단계 A)                    │
│  ─ 점검 진행 화면 (단계 B)                        │
│  ─ 양식 미리보기·인쇄 (별지 9호/4호, @page A4)    │
└──────────────┬──────────────────────────────────┘
               │ server function 직접 호출 (LLM 0)
               ↓
┌─────────────────────────────────────────────────┐
│  Domain functions (apps/web/lib/domain/)        │
│  ─ lookupLawByYear(year, lawId)                  │
│  ─ deriveInspectionScope(buildingMeta)           │
│  ─ composeWorkOrder(project)                     │
│  ─ validateInspection(inspection)                │
└──────────────┬──────────────────────────────────┘
               │
   ┌───────────┴───────────┐
   ↓                       ↓
┌─────────────────┐    ┌─────────────────────┐
│ Data (정적)      │    │ 법제처 API (단계 A만)│
│ fire-duty-master │    │ 응답은 WorkOrder에   │
│ inspection-      │    │ 동결 (작업지시=캐시) │
│   checklist      │    │                      │
└─────────────────┘    └─────────────────────┘
```

> 핵심 원칙 유지: 계산은 코드(결정론적 룰), LLM은 해석·요약만 (v1은 LLM 호출 0).
> v2 MCP 부활 경로: 위 4함수를 import하는 `apps/mcp/`를 추가하면 동일 로직을 Claude Desktop에 노출 가능.

## 5. 성공 기준 (Check 단계 판정용)

PoC v1 완료 = 다음 시나리오가 사람 손 없이 흘러간다.

- [ ] 임의 건축물(예: 2010년 사용승인, 연면적 5000㎡, 다중이용업소 일부) 입력 → 적용 소방시설 리스트가 reproducible하게 산출
- [ ] 별지 제9호서식 1쪽 ~ 8쪽 전체가 HTML로 렌더되고, **A4 인쇄 시 PDF 원본과 1:1 사이즈** (210×297mm, 단 1픽셀 오차 허용)
- [ ] 별지 제4호서식의 적용 점검표(PoC 5~7종)가 HTML로 렌더, 점검번호 체계(`{대분류}-{중분류}-{소분류}` 예: `1-A-007`) 그대로 유지
- [ ] 법제처 API 호출 실패 시 로컬 스냅샷으로 폴백, 사용자에게 폴백 사실 명시
- [ ] `fire-duty-master.json` 6종 + `inspection-checklist.json` 5~7종 모두 출처(법령 조문·페이지) 표기, 추정·환각 0
- [ ] 한국어 폼 입력 → 한국어 출력, 모든 법조문 인용은 [[2026 시행기준]] 명시
