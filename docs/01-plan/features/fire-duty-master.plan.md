# Plan: fire-duty-master (소방 법정문서 마스터 데이터)

> PDCA Plan 문서 · feature: `fire-duty-master` · 작성 2026-05-21

## 1. 배경 / 문제

소방안전점검 워크플로 시스템의 법정문서·의무 판단은 공통의 **단일 진실 공급원(SSoT)** 데이터를 참조해야 한다. 이 SSoT가 없으면 법령 근거·주기·벌칙이 화면, 룰 엔진, 출력물마다 제각각이 되어 정합성이 깨지고 환각 위험이 생긴다.

`fire-duty-master.json`은 소방 법정문서·의무를 하나의 마스터로 정의하는 데이터 자산이다.
greenfield 빌드에서 **작업량을 지배하는 핵심 산출물**이며, 이것이 먼저 서야 도구 구현이 쉬워진다.

이 feature의 마스터는 서비스가 관리하는 정적 법정문서·의무 기준이다. 건축물별 시행일에 맞춰 수집하는 법령 본문 스냅샷은 `fire-inspection-system`의 고객 BYOK 흐름에서 `law_snapshots`로 관리한다.

## 2. 목표 (이 feature의 범위)

- 소방 법정문서/의무를 표준 스키마로 정의한 `fire-duty-master.json` 설계·작성
- 각 노드: 식별자·명칭·분류·근거 법령·양식 강제성·주기·적용범위·보존기간·벌칙
- 타겟 대상물(아파트·빌딩·상가, 1~3급)에 필요한 핵심 문서를 우선 수록
- `reference/` 법령 PDF와 법제처 데이터로 각 필드를 검증 가능하게 출처 표기

### 범위 밖 (이 feature 아님)
- 등급 계산 룰 엔진 (별도 feature: assess-building-obligations)
- 웹앱 화면·API 구현 (별도 feature: fire-inspection-system)
- 건축물별 시행일 법령 스냅샷 수집·업로드 (별도 feature: fire-inspection-system)
- NFPC/NFTC 기술기준 본문 번들 (별도 데이터 작업)

## 3. 우선 수록 문서 (초안 — Design에서 확정)

| 문서 | 분류 | 근거(검증 필요) |
|------|------|------|
| 소방계획서 | 계획 | 화재예방법 |
| 소방시설 자체점검 결과보고서 (작동점검) | 점검 | 소방시설법 §22 |
| 소방시설 자체점검 결과보고서 (종합점검) | 점검 | 소방시설법 §22 |
| 소방훈련·교육 결과 기록부 | 교육/훈련 | 화재예방법 |
| 소방안전관리자 선임 신고 | 체제 | 화재예방법 |

> 우선 산출물이 "자체점검 결과보고서"이므로 점검 계열을 가장 먼저·깊게 채운다.

## 4. 문서 노드 스키마 (확정 후보)

```jsonc
{
  "docId": "self_inspection_result_report_operational",
  "title": "소방시설 자체점검 결과보고서(작동점검)",
  "category": "G_점검",
  "legalRef": ["소방시설법:22", "소방시설법시행규칙:20"],
  "formStrictness": "법령강제",       // 법령강제 | 고시 | 자율
  "frequency": "연 1회",
  "applicableScope": "특정소방대상물",
  "retention": "2년",
  "penaltyOnMissing": "소방시설법:...(과태료)",
  "source": "reference/소방시설법 및 화재예방법령집.pdf p.xxx"
}
```

## 5. 성공 기준 (Check 단계 판정용)

- [ ] 우선 수록 문서 5종 이상이 스키마 전 필드 채워짐
- [ ] 모든 `legalRef`가 reference/ 또는 법제처에서 실제 조문으로 확인됨 (출처 표기)
- [ ] 임의 추정·환각 0 — 미확인 필드는 `"TODO: 검증필요"`로 명시
- [ ] JSON 스키마 유효 (파싱 가능, 필수 필드 누락 없음)
