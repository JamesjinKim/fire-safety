Decision Queue

Phase 4 전에 고정할 정책

이 세 가지는 타입과 guard에 직접 반영되므로 뒤로 미루면 UI와 API 작업이 흔들린다.

권장안 있음

DB는 PostgreSQL 호환 RDB

핵심 관계는 정규화하고 OCR, 법령, 작업지시 동결본은 jsonb 스냅샷으로 보관한다.

권장안 있음

orgId는 모든 entity와 테이블에 포함

v1은 약간 장황하지만 v2 멀티테넌시와 데이터 접근 검증이 단순해진다.

코드화 필요

status 전이표

transitions.ts로 허용 전이를 고정하고 API 핸들러가 같은 규칙을 쓰게 한다.

임시 결정

v1 권한 모델

admin은 자기 org 전체, field는 배정된 섹션만 접근하는 guard를 우선 적용한다.

확정 방향

BYOK 법령 스냅샷

고객 관리자는 자기 법제처 API 키로 스냅샷만 만들고, 설비 리스트는 우리 룰 엔진이 생성한다.

Law Snapshot

법령 수집은 고객 키, 추천은 우리 룰 엔진

Vercel과 Supabase에는 법제처 API 키를 저장하지 않는다. 고객 관리자 세션에서 필요한 시행일 스냅샷만 생성하고, 작업지시에는 저장된 스냅샷과 룰 버전을 동결한다.

STEP 4 전

스냅샷 필요 판정

건축물대장 확정값으로 필요한 법령명, 시행일, 조문 후보를 산출한다. 없으면 추천을 보류한다.

BYOK

고객 키 기반 수집

고객 관리자가 입력한 법제처 API 키는 수집 세션에서만 사용하고 저장하지 않는다.

Snapshot

Supabase 저장

서버에는 스냅샷 JSON, 출처, 시행일, hash, 수집 시각 같은 근거 메타데이터만 저장한다.

Rule Engine

설비 추천

고객이 설비 리스트를 만들지 않는다. 우리 결정론적 룰 엔진이 적용 설비 섹션을 추천한다.

Product Flow

구현이 완성해야 할 업무 흐름

사용자 경험은 건축물대장 입력에서 현장 점검과 출력물 생성까지 끊기지 않아야 한다.

A-1 생성고객사, 건물, 점검 프로젝트를 만든다.
A-2 OCR 검수건축물대장 원본과 추출 필드를 나란히 확인한다.
A-3 설비 검수추천 설비를 포함, 제외, 수정, 보류 상태로 정리한다.
A-4 작업지시hash로 동결된 작업 범위를 만든다.
B 현장 점검설비별 결과와 불일치 기록을 입력하고 법정 서식을 출력한다.
